Windows 下通过 VS Code 远程调试 Linux C/C++ 程序配置

本文整理 Windows 主机通过 VS Code 连接 Linux 开发环境，并使用 CodeLLDB、CMake Tools 与 launch.json 完成 C/C++ 程序远程调试的基础配置流程。 这种方式适合把代码、编译器、CMake、调试器和运行环境都放在 Linux 端，而 Windows 端只保留 VS Code 图形化开发体验。

本文整理自我的原创文章，主要面向 Windows + VS Code + Linux 远程开发场景，重点保留插件安装、Linux 依赖安装、launch.json 配置、调试流程和常见问题排查。

一、配置目标

目标是在 Windows 电脑上使用 VS Code 作为开发入口，通过 SSH 打开 Linux 端工程目录，在 Linux 环境中完成 C/C++ 项目的编译与调试。 这样可以避免 Windows 与 Linux 编译环境不一致的问题，同时保留 VS Code 图形化断点、变量查看、调用栈查看等调试体验。

核心思路：Windows 端只负责运行 VS Code 客户端；代码、编译器、CMake、调试器和 VS Code Server 运行在 Linux 远程环境中。

二、VS Code 插件配置

在 VS Code 中需要准备远程连接、C/C++ 编辑与调试相关插件。对于远程 Linux 调试，建议按下面顺序配置。

  1. 在 Windows 本地 VS Code 安装 Remote - SSH，用于通过 SSH 打开 Linux 远程目录。
  2. 连接到 Linux 后，在远程环境中安装 CodeLLDB，用于 LLDB 调试。
  3. 如果项目使用 CMake，安装 CMake Tools，用于识别 CMake target，并为 ${command:cmake.launchTargetPath} 提供可执行文件路径。
  4. 建议安装 clangd 或 C/C++ 相关插件，用于代码补全、跳转和诊断。
建议重点检查三件事：VS Code 左下角是否已经连接到远程 Linux 主机；CodeLLDB 是否安装在远程环境；CMake Tools 是否已经完成 Configure 并选择了正确的 Launch Target。

三、Linux 依赖安装

在 Linux 远程主机上更新软件源并安装编译与调试工具： 

sudo apt update
sudo apt install -y build-essential cmake clangd lldb ninja-build gdb

如果是全新的 Ubuntu / Debian 环境，以上命令可以一站式安装编译和调试所需的全部常用工具，避免后续出现找不到编译器、构建工具或调试器的问题。

安装完成后，可以使用下面命令确认工具是否可用。 

cmake --version
clangd --version
lldb --version
gcc --version
g++ --version

四、launch.json 配置

在工程根目录下创建 .vscode/launch.json。如果工程由 CMake 管理，并且已经通过 CMake Tools 选择了启动目标，可以使用下面配置。

.vscode/launch.json 
{
  "version": "0.2.0",
  "configurations": [
    {
      "name": "Debug C/C++ Program",
      "type": "lldb",
      "request": "launch",
      "program": "${command:cmake.launchTargetPath}",
      "args": [],
      "cwd": "${workspaceFolder}",
      "stopOnEntry": false
    }
  ]
}

字段说明

  • name：调试配置名称，会显示在 VS Code 调试下拉菜单中。
  • type：调试器类型。使用 CodeLLDB 时填写 lldb。
  • request：调试启动模式。launch 表示启动程序并调试；attach 表示附加到已经运行的进程。
  • program：要调试的可执行文件路径。这里使用 ${command:cmake.launchTargetPath}，由 CMake Tools 自动解析当前选中的 CMake 启动目标。
  • args：程序启动参数。没有参数时保持空数组。
  • cwd：程序运行工作目录。通常设置为当前工程根目录。
  • stopOnEntry：是否在程序入口处暂停。调试普通程序时通常设置为 false。
使用 ${command:cmake.launchTargetPath} 前，需要先执行 CMake Configure，并在 CMake Tools 中选择正确的 Launch Target。否则该变量可能无法解析到可执行文件。

带参数版本

如果程序需要命令行参数，可以把参数写入 args。

.vscode/launch.json 
{
  "version": "0.2.0",
  "configurations": [
    {
      "name": "Debug C/C++ Program With Args",
      "type": "lldb",
      "request": "launch",
      "program": "${command:cmake.launchTargetPath}",
      "args": ["--config", "./config.json"],
      "cwd": "${workspaceFolder}",
      "stopOnEntry": false
    }
  ]
}

五、推荐调试流程

步骤 1：连接 Linux 主机

在 VS Code 中通过 Remote - SSH 连接 Linux 主机，并打开远程工程目录。

步骤 2：配置 CMake 工程

在 VS Code 命令面板中执行 CMake: Configure，确保工程可以正常生成构建目录。

步骤 3：选择调试目标

通过 CMake Tools 选择要运行的 target，确保 ${command:cmake.launchTargetPath} 能解析到正确的可执行文件。

步骤 4：设置断点并启动调试

在源文件中打断点，打开“运行和调试”面板，选择前面配置的 Debug C/C++ Program，点击启动调试。

六、常见问题

1. 启动调试时提示找不到可执行文件

优先检查 CMake Tools 是否已经 Configure 成功、是否选择了正确的 Launch Target，以及当前项目是否真的生成了可执行目标。

2. CodeLLDB 装了但远程调试不可用

检查 CodeLLDB 是不是安装在远程 Linux 环境中。使用 Remote - SSH 后，扩展面板里通常会区分“本地安装”和“远程安装”。

3. 断点不生效

检查程序是否以 Debug 模式编译，编译参数中是否包含调试信息。CMake 工程可以使用下面命令重新生成 Debug 构建目录。 

cmake -S . -B build -DCMAKE_BUILD_TYPE=Debug
cmake --build build

4. 不使用 CMake 怎么配置

如果程序不是 CMake 工程，可以把 program 改成固定可执行文件路径，例如： 

"program": "${workspaceFolder}/build/app"
